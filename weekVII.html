JS: obiekty, obiekty

Prawie wszystko w JS jest obiektrm*
* - nawet typy proste, mimo iż nie są obiektami, to tak się zachowują

Zmienn -> const / let
funkcje ()->{}

Obiekt jest strukturą danych {}

Przykładowy obiekt: Gwiazda śmierci

Nasz świat jest obiektowy -> przenosimy świat rzeczywisty na świat programistyczny

key: value
klucz: wartość
klucz - unikalny w ramach obiektu
wartość - prymitywy / funkcje / zagnieżdżenia

Zmienne w obiektach nazywywamy -> własnościami (properties)

funkcje w obiektach nazywamy -> metody (methods)

Poza obiektami dalej nazywamy je zmiennymi i funkcjami - odróżnienie

. - dot notation - notacja kropkowa
np. console.log(dethStar.name)
zwróci nam nazwę - dethStar

dethStar.name = 'Gwiazda śmierci
console.log(dethStar.name)
zwróci nam nazwę - Gwiazda śmierci - pozwala na zmianę wartości properties

[] - bracket notation - działa tak samo jak dot notation z tą różnicą, że można tam przywoływać properties ze spacją (jest to niepolecane, lepiej zapisywać camelCasem, ale czasem nieuniknione)
dethStar["population living"] = 0 - doda nam nową własność (property)

JSON - JavaScript Obcjet Notation
Jest to obiekt javascriptowy, zapisany w specjalny sposób i jest on przesyłany np. pomiędzy serwerem, a naszą aplikacją frontendową. Jest to uniwersalny format danych, niezależny od języka, może być wysyłany z backendu. JSON nie jest przypisany do żadnej zmiennej, musimy go przypisać
JSON - nie może posiadać funkcji; wszystko jest zapisane double casem ""

Peudoelementy & Peudoselektory

Pseudoelementy (podwójny dwukropek ::) - tworzy elementy stylujące

CSS może wpłynwać na treść

content: ""

element::before
treść elementu
element::after

Istotne - w beforze i afterze nie wrzucamy nic istotnego co może być przeczytane przez czytniki, nic co ma znaczenie pod kątekm treści, tylko rzeczy, elementy wpływające na wygląd

Pseudoklasy (pojedynczy dwukropek :) - służy do określania stanu elementu

selektor: hover - zmienia klasę, np. kolor, po najechaniu myszką
visited - odwiedzona strona zmienia kolor, własnoąć
focus - do formularzy, klasa aktywuje się po wejściu w dane pole

selektor: first-child / last-child / ... - zmienia klasę pierwszego / ostatniego elementu (element musi być pierwszy / ostatni w rodzinie - nie może być przed / po nawet zupełnie innego elementu)

selektor: not(selektor)
klasa: not(klasa--specjalna) - wszystkie elementy z daną klasą prócz klasy specjalnej dostaną daną własność

SASS

Problemy z CSS

  • porozrzucane klasy (bałagan w plikach)
  • @media oddzielone od elementu (konieczność ponownego definiowania)
  • kiepska czytelność plików (szczególnie przy zagnieżdżeniach)
  • brak możliwości definiowania funkcji do robienia podobnych rzeczy

SASS to preprocesor CSS - zamienia swoją składnię, swój kod na CSS-a, który jest rozumiany przez przeglądarkę -> SASS

SASS potrzebuje narzędzia do działania

SASS vs SCSS - są to 2 różne składnie
SASS - jest narzędziem, ma swoją składnię i ma składnię SCSS-ową (składnie podobna do pythona)
SCSS - składnia prostsza i bardziej czytelna

.scss (dla programisty) --> (SASS - narzędzie) --> .css (dla przeglądarki)

.scss - piszemy w tych plikach - narzędzie SASS zamienia je na pliki CSS - zrozumiałe dla przeglądarki

SASSMEISTER.COM - strona do ćwiczeń

Wszystko czego możemy używać w CSS-sie możemy używać w SASS-sie

Co nam daje SASS?

$zmienna
@mixin - definiuje jakieś własności
@include - wstawia tego @mixina (te własności) np.

@mixin borderBackground {
border-radious: 10 px;
background: ping;
}
.main {
@include borderBackground
}

Mixin może przyjmować parametry - $zmienne - które możemy zdefiniować w jdenym miejscu

@mixin borderBackgroundParameter ($radius, $background) {
border-radius: $radius;
background: $background;
}
.main {
@include borderBackgroundParameter (20px , red);
} - możemy definiować bezpośrednio przy elemncie

@extend - dopisuje klasy na górze (te same własności, różne klasy)

.font-style {
font-size: 13px;
colorr: green
}
.main {
@extend.font-style;
} -> spowoduje wyrzucenie do góry tych samych wartości CSS-owych w klasach

@mixin + @include - dopisuje własności do każdej klasy; SASS nie wrzuca pustych klas (większy porządek - BEM)

& parent selector - skraca zapis!

.about {
color: red;
&__title {
font-size: 18px;
&--smaller {
font-size: 16px;
}
}
}

color: red;
}

font-size: 18px;
}

Node.js & NPM

Node.js Instalacja, wybieramy wersję starszą (stabilniejsza)

Node.js utuchamiamy z terminala, komenda - node -v - sprawdza czy się zainstalował i pokazuję jaka jest wersja

Node.js to taki javascript uruchamianay poza przeglądarkąm, uruchamia się w systemie np. za pomocą terminala.
Node.js jest to środowisko, w którym piszemy za pomocą JS i możemy pisać aplikacje poza przeglądarką

Node.js nie ma dostępu do DOM

Uruchomienie pliku - node script.js

NPM nie jest tłumaczone jako node package manager (chociaż nim jest - jest to narzędzie)

Jest to taka wyszukiwarka / repozytorium z pakietami, które możemy ściągać do naszego projektu i ich używać

Po co nam te paczki?

  • dodajemy do projektów sporo gotowych bibliotek / narzędzi
  • nie można polegać na konfuguracji edytora / jego wtyczkach (każdy może mieć inne wtyczki zainstalowane lub ich nie chcieć
  • uniwersalne źródło ze wszystkimi narzędziami
  • dostęp do aktualizacji

npm to taki appstore / sklep play dla frontendowca
npm to taka skrzynka z narzędziami, których możemy użyć w naszym projekcie

npm init - jest to polecenie, które używamy tylko raz podczas zakładania świerzego projektu, podczas tworzenia możemy nadać mu nazwę itp

npm init -y - pomija pytania i tworzy od razu gotowy plik z wartościami domyślnymi

npm init -y i npm init - tworzą nam JSON-a o nazwie package.json

Przykładowa bilbioteka: moment.js - biblioteka, która formatuje czas

node_modules - jest to katalog, w którym znajdusą się paczki tzw. moduły (ten katalog szybko potrafi przybierać na wadze), tego katalogu nie edytujemy

Przywołanie komedy z biblioteki:
const moment = requier ('moment')
console.log(moment().format('MMMM Di YYYY, h:mm:ss: a'))
- komenda wykorzystująca bibliotekę moment, komendy dostępne na stronie biblioteki moment.js

Nie commitujemy katalogu node_modules !

Każdy może zrobić npm install i sobie ściągnąć wykorzystywane w projekcie paczki i w konkretnej wersji

Informacje o tym co jest (jakie paczki) wykorzystywane w projekcie znajduje się w pliku package.json -> dependencies

package-lock.json - jest to plik, który commitujemy, pozwala nam na zablokowanie wersji paczki, pilnuje czy wersja jest odpowiednia itp. Tefo pliku nie edytujemy. Zwiększa on bezpieczeństwo w naszym projekcie - instalowana będzie dokładnie ta wersja paczki, którą użyliśmy tworząc projekt, nie ma możliwości, że wyjdzie nowa wersja, zastąpi użytą i coś przestanie działać

package.json można edytować i commitować
package-lock.json nie można edytować ale commitujemy

WTF GULP STARTER KIT

.git clone https://github.com/maciejkorsan/wtf-gulp-starter - klonowanie startera z githuba

GULP - jest to narzędzie, które pozwala na tworzenie tzw. tasków (zadań), które są wykonywane. Można powiedzieć, że jest to uruchamiacz zadań - GULP

WTF GULP STARTER KIT - co się w nim znajduje

  • serwer lokalny z autoodświerzaniem - jeśli zmienimy coś w pliku i zapiszemy, to przeglądarka automatycznie się przeładuje (nie trzeba odświeżać), działa również na telefonie w lokalnej sieci
  • wsparcie dla scss i sourcemap
  • minifikacja - zmniejszenie kodu, żeby jak najmniej ważył
  • webpack
  • publikacja na github pages za pomocą polecenia z naszego komputera

gulpfile.habel.js - plik z taskami i innymi

assets - katalog zawierający pliki ogólne: zdjęcia, filmy itp.

Instalacja gulpa

gulpa instalujemy globalnie - jako administrator, raz na cały system

cmd -> ppm -> uruchom jako administrator -> npm install -g gulp-cli

npm install (tylko raz na projekt i w momencie gdy ściągniemy nową wersję lub dokładamy coś od siebie)

gulp - komenda w terminalu, którą się uruchamia

ctrl + c - wyjście z programu w terminalu do terminalu

local: httpl://localhost:3000 - nasz serwer lokalny
external: http://192.168.8.101:3000 - link do naszej strony postawionej na serwerze lokalnym, na telefonie też można otwierać (działa tylko w obrębie jednej sieci wewnętrznej)

src/content/plik.html -> dist/plik.html

Katalog SRC (source) to źródło naszej strony. Tutaj pracujemy, edytujemy itp.
To co jest w katalogu src gulp kopiuje i przetwarza do katalogu dist
Dista nie edytujemy - po zapisaniu wszystkie zmiany zostaną utracone (zostanie i tak przetworzony katalog src), jest to projekt przekompilowany i gotowy do publikacji - wszystko co zmienimy w dist zostanie nadpisane z katalogu src
Dist jest źródłem serwera
Katalog dist możemy nawet usunąć (np. żeby go oczyscić - w przypadku usunięcia np. zdjęć), po ponownym odpaleniu gulpa utworzy się na nowo
Dista nie trzeba commitować, można go dodać do .gitignore

Pracujemy w src np. na scss-ach. Gulp to przerobi na css-y i utworzy w dist - będzie to możliwe do otworzenia w przeglądarkach, serwer pobierze te pliki (z dista)

src/scss/main.scss -> dist/css/main.css

@import "normalize" - pobiera nam inne pliki scss-owe np. pobierze normalize do main.css

Normalize, zmienne, breakpointy i inne powinny być w osobnych plikach i @importowane

sourcemap - pokazuje np. w devtoolsach gdzie dana własność występuje np. SCSS 9 linijka - jest to zapisane w main.css.map (niezrozumiałe dla człowieka - nie trzeba się w to zagłebiać)

src/assets/img/plik.png -> dist/assets/img/plik.png

Nie edytujemy katalogu dist! - żadne zmiany nie zostaną zapisane (chyba że taki jest nasz cel)

DIST jest wynikiem naszej pracy, zawartość tego katalogu publikujemy na serwerze

npm run deploy - npm script - wchodzi w skrypty i je odpala
tworzy nowy branch na githubie (gh-pages), tworzy stronę tylko do publikacji i tylko z katalogu DIST (może się różnić od commitów)

paczka gh-pages - tworzy skrypt, który pomaga publikować strony w package.json

"scripts": {
"deploy": "hg-pages-d-dist"
} - git hub pages będzie pobierał katalog dist

Kolejność działania

Kolejność działania

  1. Z poziomu terminala pobieramy starter kit-a - git clone link do repozytorium Korsana
    • tworzy nam folder wtf-gulp-starter
  2. Tworzenie nowego projektu
    • nowy folder np. homepage-gulp
    • kopiujemy zawartość z wtf-gulp-starter do nowego projektu
  3. Tworzymy nowe repozytorium z terminala lub git huba
    • kopiujemy komenty do gita z git huba, wypisane są podczas tworzenia repozytorium - komendy uruchamiające wszystko np. git init, git add README
    • wklejamy powyższe komendy do terminala w naszym projekcie
    • git commit -m "initial commit"
  4. Instalacja gulp-a globalnie (raz na system, jako administrator) - npm install -g gulp-cli
  5. npm install - instalacja, do projektu, paczek ze startera
  6. komenda gulp - uruchamia projekt (za każdym razem gdy zaczynamy pracę) - odpala serwer lokalny
  7. npm run deploy - publikacja na gh-pages
